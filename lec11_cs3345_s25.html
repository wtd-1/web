Introduction

Until this point, we have been discussing how we can utilise different data structures to make our jobs more efficient. That said, even more efficient data structures such as linked lists can be further optimised. Consider a ginormous list of size \(n\). If we want to search for an element in this list, we would have to traverse the entire list to find the element, which takes \(O(n)\) time. That is not bad, but it can be must better. Trees are a data structure that can help us search, and do most of our arsenal of operations, for elements in \(O(\log n)\) time. This is a significant improvement over \(O(n)\) time.

Trees

A tree is a collection of nodes, consisting of a node \( r \) called the root and zero or more nonempty subtrees \( T_1, T_2, \dots, T_k \), each of whose roots are connected by a direct edge from \( r \).

Paddy3118, CC BY-SA 4.0, via Wikimedia Commons

In the figure above, we see that the node in the red colour is the root of the tree, which is a parent of 7 and 5. From there, 7 and 5 are themselves parents—the former to 2, 10, and 6; the latter to just 9. After that, 6 is a parent to 5 and 11 and 9 is a parent to 4. All of these nodes share a common ancestor: 2

Incidentally, we can generalise trees to be comprised of \(n\) nodes. How many edges do you think a tree has? \(n - 1\)!

Terminology

Consider the figure below.

Paddy3118, CC BY-SA 4.0, via Wikimedia Commons, edited by William T. Doan

The figure above is to help you visualise some of the definitions that I will rattle off below.


Path. A path from node \( n_1 \) to \( n_k \) is defined as a sequence of nodes \( n_1, n_2, \dots, n_k \), such that \( n_i \) is the parent of \( n_{i+1} \) for \( 1 \leq i \leq k \). The length of the path is \( k - 1 \).

Length of Path. The number of edges on the path.

Depth of Node \( n_i \). The length of the path from the root to \( n_i \).

Tree Depth. The length from the root to the deepest leaf (or whatever you are looking for), also known as tree height.

Height of Node \( n_i \). The length of the longest path from \( n_i \) (i.e., some arbitrary node) to the deepest leaf along the path formed.

Depth of Root. The depth of the root is 0, and the height of all leaves is 0.

Edge. A connection between one node and another.

Path. A sequence of nodes and edges connecting a node with a descendant.

Height of Node. The height of a node is the number of edges on the longest downward path between that node and a leaf.

Height of Tree. The height of a tree is the number of edges on the longest downward path between the root and a leaf. The height of a tree is equivalent to the height of its root.

Maximum Number of Nodes in a Tree of Height \( h \). A tree with height \( h \) has at most \( 2^h - 1 \) nodes.

Depth. The depth of a node is the number of edges from the node to the tree's root node.

Level. The level of a node is defined as \( 1 + \) the number of connections between the node and the root.

Application of Trees: Directories

I am fairly certain any person at UTD has access to a computer, be it at home, the library, or the computer lab. Have you ever wondered how files and folders are organized? Well, in an operating system, the file system is structured as a tree, where directories act as nodes and files are stored within them.

The root directory is the starting point of the file system hierarchy, much like the root of a tree. Each directory may contain subdirectories, forming a branching structure that mirrors a tree’s organization. This structure allows for efficient file retrieval, as paths to files can be traversed in a structured manner.

Each file or directory in a system has a unique path, which can be either absolute (starting from the root) or relative (starting from the current working directory). Operations such as creating, deleting, or moving files involve traversing this hierarchical tree structure.

Peter Flass, CC BY-SA 4.0, via Wikipedia

We can use the pseudocode below to list a directory in a hierarchical system.


    private void lisstALl( int depth ) {

        printName( depth ); // Print the name of the current directory
        
        if ( isDirectory() ) {
            for ( each entry in the directory ) {
                entry.listAll( depth + 1 );
            }
        }

    }

    public void listAll( ) {

        listAll( 0 );

    }

From Application to Content

It turns out that thinking about trees in terms of directories, which you will see when you take Unix, is an intuitive way to explain trees and its operations. One such operation is Traversal. There are three ways to do so:

  • Preorder. You go from the parent first, then the children.
  • Postorder. You go from children first, and then the parent.
  • Inorder. You go from the left child, then the parent, and then the right child. Imagine a triangle—you start off at the left corner, go to the apex, and then end at the right corner.
    • This is actually especially important for binary trees!

Directory structures can be implemented as trees, and it makes sense to print a directory in preorder. Preorder prints the parent before printing its children, thus printing the directory name before printing the files it contains That is why when you open a folder, you can access all those subfolders therein. We might traverse a directory in postorder if we wished to sum file sizes and roll these up to the parent levels.

Now, what if I wanted to print the size of the directory? Well, that means you cna use the Postorder Traversal technique. Look at the pseudocode below.


    public int size() {

        int total = sizeOfEntry();

        if ( isDirectory() ) {
            for ( each entry in the directory ) {
                total += entry.size();
            }
        }

        return total;
    }

Binary Trees

Remember when I said that trees on their own are a little boring? I think I mentioned something to that effect at the beginning (I think). Here is when things get interesting! We will now begin talking about Binary Trees, which are trees where each node has at most two children. This is a very important data structure, and it is used in many applications.

Properties of a Binary Tree

A binary tree is said to:

  • Have a depth of \(O(\sqrt{n})\), which means the height of the tree grows proportionally to the square root of the number of nodes.
    • Though, there is a special case! In Binary SEARCH Trees (NOT to be mistaken for binary trees!), the average depth is \(O(logn)\) (the "average" part will be important at some point)
  • Be as large as \(n - 1\)

Looking at the figure above from lecture, we can recognise the familar three traversal methods and how they are applied to the binary tree.

  • Postorder. We go left, right, and then to the node

  • Preorder. We go to the node first, then the left node, and then the right node
      • Example : + + a * b c * + * d e f g

Constructing an Expression Tree

An algorithm to convert a postfix expression into an expression tree follows a structured approach using a stack. The expression is processed one symbol at a time. If the symbol is an operand, it is pushed onto the stack. If the symbol is an operator, two trees are popped from the stack to create a new tree where the operator becomes the root, and the two popped trees serve as the left and right children. The newly formed tree is then pushed back onto the stack until the entire expression is processed. 


    Algorithm postfixToExpressionTree(expression)

    Input: A postfix expression
    Output: An expression tree representing the given postfix expression

    Procedure:
        1. Create an empty stack S
        2. For each symbol in the expression:
        3.     If the symbol is an operand:
        4.         Create a new tree node with the symbol
        5.         Push the node onto stack S
        6.     Else if the symbol is an operator:
        7.         Pop T1 ← S.pop()
        8.         Pop T2 ← S.pop()
        9.         Create a new tree node with the operator as the root
        10.        Set T2 as the left child of the new node
        11.        Set T1 as the right child of the new node
        12.        Push the new node onto stack S
        13. Return the remaining node in S as the root of the expression tree

The Search Tree ADT

Remembe when I mentioned how you cannot mistake a binary tree and a binary search tree? Well, the property that makes an binary tree into a binary search tree is that for every node \(X\):

  1. the value of all the items in its left subtree are smaller than \(X\)
  2. the value of all the items in its right subtree are larger than \(X\)

Implementing the Search Tree ADT via the BinaryNode Class


    private static class BinaryNode<AnyType> {

        BinaryNode( AnyType theElement ) {

            this( theElement, null, null );

        }

        BinaryNode( AnyType theElement, BinaryNode<AnyType> lt, BinaryNode<AnyType> rt ) {

            element = theElement;
            left = lt;
            right = rt;

        }

        AnyType element;
        BinaryNode<AnyType> left;
        BinaryNode<AnyType> right;

    }

A Specific Glance: the Contains Method


    public boolean contains( AnyType x, BinaryNode<AnyType> t ) {

        if ( t == null ) {
            return false;
        }

        int compareResult = x.compareTo( t.element );

        if( compareResult < 0 ) {
            return contains( x, t.left );
        } else if( compareResult > 0 ) {
            return contains( x, t.right );
        } else {
            return true
        }

    }

The code above is the contains operation for binary search trees. It is an internal method that allows us to find an item in a subtree given the thing we want to find, \(x\), and the root of the subtree you think it's in, \(t\). Since we are trying to find osomething, we return true once we've found it.

What we are doing here is taking \(x\) and comparing it with \(t\). In case you don't already know, compareTo takes an object, in this case \(x\) and compares it with the specified object, in this case \(t\), for order; it returns a negative integer, zero, or a positive integer as this object is less than, equal to, or greater than the specified object respectively. That means if \(x\) is less than \(t\), we now to look at the left subtree; conversely, if \(x\) is bigger than \(t\), we go to the right subtree. We keep doing so until we have found what we are looking for.

I wanted to bring your attention to the methodology we used, comparing \(x\) and \(t\). Why would we do that? Hint: look at The Search Tree ADT section.

Another Specific Glance: findMin and findMax

To perform a findMin, start at the root and go left as long as there is a left child. The stopping point is the smallest element. findMin is done recursively and findMax is done using a while loop (i.e., nonrecursively). 


    // Implementation of findMin
    private BinaryNode<AnyType> findMin( BinaryNode<AnyType> t ) {

        if ( t == null ) {
            return null;
        } else if ( t.left == null ) {
            return t;
        }

        return findMin( t.left );

    }



    // Implementation of findMax
    private BinaryNode<AnyType> findMax( BinaryNode<AnyType> t ) {

        if ( t != null ) {
            while ( t.right != null ) {
                t = t.right;
            }
        }

        return t;

    }

Yet Another Specific Glance: insert

To insert \(X\) into the tree \(T\), proceed down the tree as you would with a contains. If \(X\) is found, do nothing.

    private BinaryNode<AnyType> insert( AnyType x, BinaryNode<AnyType> t ) {

        if ( t == null ) {
            return new BinaryNode<>( x, null, null );
        }

        int compareResult = x.compareTo( t.element );

        if( compareResult < 0 ) {
            t.left = insert( x, t.left );
        } else if( compareResult > 0 ) {
            t.right = insert( x, t.right );
        } 

        return t;

    }

A Final Glance: remove

Deletion is fun, but it's hard.

  • Case 1. The node to be deleted is a leaf node. In this case, we can simply remove the node from the tree.
  • Case 2. The node to be deleted has one child. In this case, we can replace the node with its child.
  • Case 3. The node to be deleted has two children. In this case, we can replace the node with the smallest node in the right subtree.
Three cases?! That's a lot to consider...which is why Lazy Deletion is a popular strategy. The premise is simple: when an element is to deleted, it is left in the tree and merely marked as being deleted. Take a look at the figures below.

Image from Meghana Navanth Satpute, edited by William T. Doan, CC BY-SA 4.0, via The University of Texas at Dallas

What we did to “delete” the node (4) is remove its connection to the tree. After that, we connect the former parent node of node (4) to node (3), its new child. Do you see the Lazy Deletion here? We left the node (4) in the tree, but it has been marked as “removed”.

Now, here is an example where we remove a node with not one, but two children.

Image from Meghana Navanth Satpute, edited by William T. Doan, CC BY-SA 4.0, via The University of Texas at Dallas

Notice what happened to the node three! We wanted to delete node (2), and we did. Let's take a closer look at what that means intermediately.

Images from Meghana Navanth Satpute, edited by William T. Doan, CC BY-SA 4.0, via The University of Texas at Dallas

Let us start with part one. We see that we wanted to remove node (2). Great! It's gone. Now, we have an empty space that needs to be filled. What do we fill it with? Well, we have all the nodes we didn't delete—so we need to fill the space with one of those nodes. We can't just pick any node, though. We need to pick the smallest node such that the tree remains a binary search tree. What is that node? It's node (3)! So, we replace node (2) with node (3).

Now, let's move on to part two. Now that node (3) has been moved, we have node (4) left. What do we do with it? We have to keep it in the tree, because it's one of the original nodes we had. How can we keep node (4) in the tree while also respecting it's properties? Easy: connect it with node (5)! It's greater than node (3), so it goes to the right subtree, where node (5) is. However, it is less than node (5), so we plant node (4) in the left subtree of node (5). And there you have it! We have successfully removed node (2) and kept the tree a binary search tree.

With that out of the way, here's the code for the delete routine for a binary search tree. 


    private BinaryNode<AnyType> remove( AnyType x, BinaryNode<AnyType> t ) {

        if( t == null ) {
            return t;
        }

        int compareResult = x.compareTo( t.element );

        if( compareResult < 0 ) {
            t.left = remove( x, t.left );
        } else if ( compareResult > 0 ) {
            t.right = remove( x, t.right );
        } else if ( t.left != null && t.right != null ) { // This is the case of two children
            t.element = findMin( t.right ).element;
            t.right = remove( t.element, t.right );
        } else {
            t = ( t.left != null ) ? t.left : t.right;
        }

        return t;

    }

Average Runtime Analysis in BST

The average running time of all the operations discussed in the previous section is \( O(\log N) \). However, this is not entirely accurate in all cases.

One of the main reasons for this discrepancy is the imbalance that can occur during deletions. When deleting nodes in a Binary Search Tree (BST), the removal process tends to make the left subtree deeper than the right. This happens because when replacing a deleted node, the common approach is to use either the in-order predecessor (largest node in the left subtree) or the in-order successor (smallest node in the right subtree). In many cases, the in-order predecessor is chosen, which can lead to an increase in depth on the left side over multiple deletions.

As a result, if a BST experiences frequent deletions, it may become unbalanced, causing the height to deviate from \( O(\log N) \) and approach \( O(N) \) in the worst case. This imbalance affects the average runtime of operations such as search, insertion, and deletion, making them slower than the expected \( O(\log N) \).

To mitigate this issue, self-balancing BSTs like AVL trees and Red-Black trees enforce balance conditions to maintain \( O(\log N) \) performance across all operations, regardless of insertions or deletions. But, I'm getting ahead of myself here...
